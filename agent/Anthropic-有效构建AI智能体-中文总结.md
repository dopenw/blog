# 《Building Effective Agents》中文总结

> 原文：Anthropic Engineering，[Building effective agents](https://www.anthropic.com/engineering/building-effective-agents)（2024-12-19）  
> 本文为归纳性中文总结，不是逐句翻译。

## 一句话结论

构建 LLM 智能体的关键不是堆叠复杂框架，而是从**最简单、可评估的方案**开始；只有在复杂度能被数据证明带来收益时，才逐步加入工作流或自主智能体。

## 1. 先区分：工作流与智能体

| 类型 | 控制方式 | 优点 | 适用场景 |
| --- | --- | --- | --- |
| **工作流（Workflow）** | 由预先编写的代码决定模型调用顺序和工具路径 | 可预测、一致、容易调试 | 任务步骤较固定、可清楚拆分 |
| **智能体（Agent）** | 由模型根据当前状态动态决定计划与工具调用 | 灵活，可处理开放式问题 | 无法预知步骤数或固定执行路径 |

二者都属于“智能体式系统（agentic systems）”。选择原则是：若单次 LLM 调用配合检索、上下文示例已能满足需求，就不必引入多步骤系统。更复杂的系统通常以更高的延迟和成本换取更好的任务表现。

## 2. 基础单元：增强型 LLM

所有模式的核心都是一个可被增强的 LLM。常见增强能力包括：

- **检索**：查找外部知识或文档；
- **工具**：调用 API、搜索、执行代码、编辑文件等；
- **记忆**：保存与任务相关的状态或历史信息。

重点不在“工具越多越好”，而在于这些能力是否贴合具体业务，并且是否为模型提供了清晰、完善、易调用的接口。

## 3. 五种常见工作流模式

### 3.1 提示链（Prompt Chaining）

将大任务拆成固定的串行子任务；前一步输出成为后一步输入。可在中间插入程序化校验关卡（gate）。

**适合：** 能够清晰拆成固定步骤，且宁可增加一点延迟来换取准确率的任务。

**例子：** 先生成营销文案，再翻译；先生成文章大纲，校验其是否符合要求，再据此撰写正文。

### 3.2 路由（Routing）

先对输入分类，再交给相应的专用提示、模型、工具或下游流程处理。

**适合：** 输入存在清晰类别，且分类足够可靠的任务。

**价值：** 避免一个通用提示同时兼顾所有场景，导致优化某类请求时损害另一些请求；也可把简单任务交给更低成本模型、复杂任务交给能力更强模型。

**例子：** 将客服请求分流为咨询、退款和技术支持。

### 3.3 并行化（Parallelization）

多个 LLM 调用同时运行，并由程序汇总结果。常见有两种形式：

- **分区（Sectioning）**：把彼此独立的子任务并发处理，以缩短总耗时；
- **投票（Voting）**：让多个调用解决同一个问题，再用投票或聚合提高置信度。

**适合：** 子任务能够独立并行，或需要多视角、多次尝试来提高可靠性。

**例子：** 分别处理用户问题与内容安全检查；使用多个提示审查代码漏洞。

### 3.4 编排者—执行者（Orchestrator–Workers）

中心模型（编排者）根据输入动态拆解任务、分派给多个工作模型（执行者），最后整合结果。

**适合：** 任务很复杂，所需子任务及其数量无法提前规定。

**与并行化的区别：** 并行化通常预先定义好任务划分；该模式中的划分由编排者在运行时决定。

**例子：** 一次代码需求会影响哪些文件、每个文件怎么改都因需求而异；多来源检索与分析任务。

### 3.5 评估者—优化者（Evaluator–Optimizer）

一个模型产出初稿，另一个模型评估并给出反馈；产出模型据此迭代，直到达到要求或触发停止条件。

**适合：** 有明确评估标准，且迭代修订能显著改善结果的任务。

**判断信号：** 人类能清楚说明怎样改进答案，并且模型也能提出类似的有效反馈。

**例子：** 文学翻译的反复润色；需要多轮检索才能完成的复杂研究任务。

## 4. 何时应该使用自主智能体

自主智能体以“模型调用工具—观察环境反馈—继续决策”的循环完成任务。它适用于：

- 任务开放、步骤数不可预测；
- 很难为任务硬编码固定路径；
- 可以在可信或隔离的环境中允许模型连续操作；
- 能从工具结果、测试结果等环境事实（ground truth）持续校正自身；
- 有明确的完成条件、迭代上限和必要的人类检查点。

其代价是成本更高、耗时更长，并可能累积错误。因此应在沙箱环境充分测试，并配置护栏（guardrails）和停止条件。

## 5. 工具设计就是“给工具做提示工程”

工具定义不只是 API 文档，而是模型行动能力的一部分，应像系统提示一样认真设计。

### 设计原则

1. **让模型在行动前有足够空间思考**，避免过早陷入难以修正的格式或决策。
2. **采用模型熟悉的自然格式**，尽可能贴近常见文本、代码或命令形式。
3. **消除无意义的格式负担**，例如要求模型手工维护大量行号、在 JSON 中转义大段代码。
4. **将工具描述写得像给初级开发者的优秀 docstring**：交代用途、参数、示例、边界、异常情况，以及与相似工具的区别。
5. **用真实样例反复测试模型如何调用工具**，根据错误调整命名、参数和描述。
6. **防错设计（poka-yoke）**：通过参数和约束让错误更难发生。

文中案例：代码智能体在切换工作目录后容易因相对路径出错；改为强制使用绝对路径后，工具调用显著稳定。

## 6. 框架的定位

框架可以降低工具调用、结构化解析、调用编排等入门成本；但也会增加抽象层，使提示和实际响应更难观察、调试，且容易诱发不必要的复杂设计。

建议是：

- 先直接使用 LLM API 实现原型；
- 若使用框架，仍要理解其底层调用和提示；
- 进入生产后，必要时减少抽象、回归基础组件，以获得更强的可控性和可维护性。

## 7. 典型高价值场景

### 客服

客服既需要多轮对话，也需要查询客户数据、订单与知识库，甚至执行退款、更新工单等操作。它通常有可衡量的解决标准，适合在人工监督下使用工具型智能体。

### 编码

代码可以由自动化测试验证；智能体可依据测试反馈持续修改；问题空间相对结构化，质量也较容易客观评估。因此编码是智能体的重要落地领域，但自动测试不能替代人工对整体架构和业务需求的审查。

## 8. 可执行的落地路线

```text
单次 LLM 调用 + 检索/上下文示例
        │（评估显示效果不足）
        ▼
固定任务 → 选择提示链、路由、并行化、评估-优化等工作流
        │（步骤和路径无法预先定义）
        ▼
引入工具循环的自主智能体
        │
        ▼
用评测、沙箱、护栏、停止条件和人工检查持续控制风险
```

每一次升级都应由评测证据驱动，而非因为“智能体架构看起来更先进”。

## 9. 最终原则

1. **保持简单**：使用刚好足够的架构。
2. **保持透明**：让规划、状态与关键决策可见、可调试。
3. **认真设计 Agent–Computer Interface（ACI）**：把模型使用工具的体验，当作人机交互一样重视。

> 成功不在于构建最复杂的系统，而在于为具体问题构建正确、可靠、可维护且值得信任的系统。
