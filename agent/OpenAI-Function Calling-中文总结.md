# 《Function calling》中文总结

> 原文：OpenAI Developers，[Function calling](https://developers.openai.com/api/docs/guides/function-calling)  
> 本文为归纳性中文总结，不是逐句翻译。内容依据 2026-07-11 阅读的官方文档整理。

## 一句话结论

**Function Calling（也称 Tool Calling）让模型决定何时请求使用应用提供的能力，但真正执行工具、校验权限与把结果回传模型的责任始终在应用侧。** 可靠实现的关键是：清晰的工具描述、受约束的 JSON Schema、严格的服务端校验，以及完整地运行“模型请求 → 应用执行 → 结果回传”的循环。

## 1. 它解决什么问题

模型本身不能直接访问你的实时数据或外部系统。通过在请求中声明工具，应用可以让模型请求执行查询或动作，例如：

- 查询某城市当前天气；
- 按用户 ID 获取账户信息；
- 为丢失订单发起退款；
- 调用内部 CRM、库存、工单或业务 API。

工具调用不是模型在服务器上自动运行函数：模型只会返回**要调用的工具名和参数**；应用负责路由、认证、授权、执行、错误处理与审计。

## 2. 基本术语

| 术语 | 含义 |
| --- | --- |
| **工具（tool / function）** | 应用告知模型可使用的一项外部能力。函数工具以 JSON Schema 描述输入。 |
| **工具调用（tool call）** | 模型请求使用某个工具，并给出 JSON 编码的参数。 |
| **工具输出（tool output）** | 应用执行工具后的结果，带着对应调用 ID 回传给模型。 |
| **函数工具（function tool）** | 输入由 JSON Schema 约束，适合有明确字段的 API 或业务动作。 |
| **自定义工具（custom tool）** | 工具输入是任意文本；可选用语法约束文本格式，适合代码、表达式等非 JSON 输入。 |

## 3. 工具调用的完整闭环

```text
用户请求 + 可用工具定义
            │
            ▼
        模型生成
            │
            ▼
  function_call（名称 + 参数）
            │
            ▼
应用校验、鉴权并执行真实函数
            │
            ▼
function_call_output（绑定 call_id）
            │
            ▼
  再次请求模型，得到最终答复或下一轮调用
```

官方将流程概括为五步：

1. 向模型发送请求，并附上可调用工具；
2. 接收模型返回的工具调用；
3. 在应用侧使用调用参数执行代码；
4. 将工具输出发回模型；
5. 接收最终回复，或继续处理更多工具调用。

Responses API 可以持续执行这一循环，直至任务完成。对推理模型，若响应中同时包含推理项和工具调用，回传工具输出时还必须保留这些推理项，以维持模型所需的上下文。

## 4. 如何定义一个函数工具

每个函数工具通常放在请求的 `tools` 参数中，核心字段如下：

| 字段 | 作用 |
| --- | --- |
| `type` | 固定为 `"function"`。 |
| `name` | 函数名，例如 `get_weather`。应稳定、明确且能反映动作。 |
| `description` | 说明何时调用、完成什么工作、参数期望是什么。它直接影响模型的选工具和填参质量。 |
| `parameters` | 描述输入参数的 JSON Schema。可使用对象、数组、枚举、嵌套对象、描述等能力。 |
| `strict` | 是否启用严格模式，使调用遵循 Schema。 |

一个典型的天气工具可把 `location` 和 `units` 定义为对象属性，用 `enum` 约束单位取值，并在 `required` 中列出所需字段。**Schema 不只是接口文档，也是模型生成参数时的约束与提示。**

### 工具设计建议

- 名称应是具体动作，如 `get_order_status`、`create_refund`，避免 `process_data` 一类泛称。
- `description` 要说明适用边界、重要前置条件和参数语义；相似工具尤其要写清差异。
- 参数应尽量小、明确、可验证；使用枚举替代开放字符串。
- 相关工具可用命名空间分组，例如 `crm.search_customer`、`billing.create_refund`、`shipping.track_package`，帮助模型区分不同业务域。
- 工具多或 Schema 很大时，可使用 Tool Search 延迟加载少用工具，减少每轮传入的工具负担；官方文档注明该能力要求 `gpt-5.4` 或更高版本模型。

## 5. 接收、执行并回传调用

模型一次响应可能包含 **0 个、1 个或多个**调用，因此实现时应遍历所有调用，而不是假设只会有一个。

### Chat Completions API

- 从响应的 `tool_calls` 数组读取调用；每项包含 `id`、函数 `name` 和 JSON 编码的 `arguments`。
- 解析参数后，仅通过自己的白名单路由到允许执行的函数。
- 把执行结果作为 `role: "tool"` 消息加入上下文，并使用 `tool_call_id` 关联原调用。
- 再向模型发起请求，让它根据结果生成面向用户的回复。

### Responses API

- 在 `response.output` 中筛选 `type: "function_call"` 的项；使用其中的 `name`、`arguments` 和 `call_id`。
- 执行后向下一轮 `input` 添加 `type: "function_call_output"`、相同 `call_id` 与 `output`。
- 继续调用 `responses.create`，直到不再有需要处理的函数调用。

工具输出通常是字符串，格式可自行定义（JSON、错误码或纯文本）；返回图片或文件的工具也可回传相应对象数组。没有业务返回值的动作类工具应至少回传明确成功/失败状态，例如 `"success"`。

### 必须放在应用侧的安全控制

即使启用了严格模式，也不能把模型输出直接执行。应用仍应：

- 校验 JSON 是否符合预期类型与业务规则；
- 使用工具名白名单，拒绝未知路由；
- 以当前登录用户而非模型身份完成认证与授权；
- 对退款、发信、删除、支付等高风险动作加入确认、限额、幂等与审计；
- 将工具异常转化为结构化错误结果回传模型，而不是让流程崩溃。

## 6. 调用行为控制

默认情况下，模型自行决定是否调用工具及调用次数；可以通过 `tool_choice` 调整：

| 配置 | 含义 | 适合场景 |
| --- | --- | --- |
| `"auto"` | 默认值；可调用 0、1 或多个工具。 | 通用问答与自主编排。 |
| `"required"` | 要求至少调用一个工具。 | 必须从实时/可信数据源取数。 |
| 指定函数 | 强制调用某一个函数。 | 流程已确定，只需要模型抽取参数。 |
| `allowed_tools` | 从已定义工具中限制当轮可调用的子集。 | 保留稳定工具列表以利用提示缓存，同时收紧当轮权限。 |
| `"none"` | 禁止调用工具。 | 只希望模型基于现有上下文回答。 |

模型可能并行请求多个函数。如业务流程要求严格串行，设置 `parallel_tool_calls: false`，这样每轮只会有 0 或 1 个调用。即便允许并行，也应分别校验、执行和关联每个调用结果。

## 7. 严格模式（Strict Mode）

官方建议在函数工具中启用 `strict: true`，使模型调用可靠遵循 Schema，而不是“尽力而为”。它建立在 Structured Outputs 之上，但 Schema 必须满足额外限制：

1. `parameters` 中每一层对象都要设置 `additionalProperties: false`；
2. `properties` 中的所有字段都必须出现在 `required`；
3. 逻辑上的可选字段可用联合类型包含 `null`，例如 `"type": ["string", "null"]`，而不是从 `required` 移除。

若 `strict: true` 但 Schema 不合规，请求会被拒绝并返回缺失约束的提示。Responses API 会尽可能把未显式设置的 Schema 规范化为严格模式；无法兼容时才退回非严格的最佳努力模式，并在响应工具中标识 `strict: false`。Chat Completions API 默认仍是非严格模式；若要在 Responses API 中明确采用非严格行为，可设置 `strict: false`。

严格模式并不支持 JSON Schema 的全部特性。对于微调模型，Schema 首次使用会被处理并缓存；频繁变动 Schema 可能增加延迟，而且这些缓存不符合零数据保留资格。

## 8. 流式工具调用

打开 `stream: true` 后，工具调用参数会以增量片段返回，应用可以及时展示“正在查询天气”等进度。处理方式与普通文本流式输出相似，但需要按调用的 `index` 聚合 `arguments` 片段：

- 首个增量通常提供 `id`、函数 `name`、`type`；
- 后续增量主要补充 JSON 参数字符串；
- 聚合完成、JSON 完整且通过验证后再执行工具；
- Responses API 还会发出 `response.function_call_arguments.done`，其中包含完整调用项。

不要在参数 JSON 尚未完整时执行写操作；流式输出适合展示进度，不应绕过完整校验流程。

## 9. Custom Tools 与 CFG

当工具不适合用字段化 JSON 表达时，可以声明 `type: "custom"`。模型会为该工具返回任意文本输入，例如一段 Python 代码或数学表达式；工具输出仍按调用 ID 回传。

可通过 `format` 为自定义工具添加上下文无关文法（CFG），约束模型输出文本。当前支持：

- `lark`：适用于带有规则与终结符的结构化语言；
- `regex`：适用于单行、固定模式的文本。

文法/正则使用 Rust regex 语法，不是 Python `re`；均不支持前瞻/后顾和懒惰量词。Lark 还不支持终结符优先级、模板、除内置 `common` 之外的导入和 `%declare`。

### 文法实践要点

- 保持文法短小、显式、边界清楚；过于复杂可能被 API 拒绝。
- 大写终结符负责词法切分，小写规则负责组合已界定的 token。
- 表达自由文本中的固定锚点时，使用一个有界的终结符/正则，不要把自由文本拆散到多个规则间；词法分析先于语法规则并采用贪婪匹配。
- 明确处理空白字符，避免无限制的 `%ignore`。
- 若出现超长、重复但语法合法的输出，收紧文法，补充少样本示例和工具说明，必要时提高推理强度。

## 10. 推荐实现清单

```text
1. 只声明任务真正需要的工具，并写清名称、描述和输入 Schema
2. 为函数工具启用 strict: true，按严格模式规则编写 Schema
3. 请求模型；遍历所有 function call
4. 在服务端做白名单、参数、认证、授权与业务校验
5. 执行工具，将成功或失败结果与正确的 call_id 回传
6. 保留必要上下文并继续循环，直至模型给出最终答复
7. 对写操作增加确认、幂等、审计和人工兜底
8. 以真实场景测试：缺参、错误参数、多工具、并行、超时和失败重试
```

## 最终要点

- Function Calling 把模型从“生成文本”扩展为“能够请求外部能力”，但模型不是权限主体，也不会替你执行函数。
- 成功率很大程度取决于工具 API 的产品化程度：清晰命名、好描述、少歧义参数与严格 Schema。
- 严格模式能提升参数结构可靠性，却不能取代服务端鉴权、业务校验和高风险操作控制。
- 把它当作一个可循环的协议来实现：**接收调用、执行、回传、继续推理**，而不是一次 API 请求。
- Custom Tools + CFG 适合特殊文本输入；优先使用简单、可验证的文法，避免把复杂语法当作业务安全机制。
