# 《A Practical Guide to Building Agents》中文总结

> 原文：OpenAI，[A practical guide to building agents](https://openai.com/business/guides-and-resources/a-practical-guide-to-building-ai-agents/)  
> 本文为归纳性中文总结，不是逐句翻译。

## 一句话结论

智能体用于替用户完成端到端、多步骤且带有不确定性的工作流。可靠落地应从**单智能体 + 明确工具 + 清晰指令 + 分层护栏**开始，以评测和真实反馈驱动迭代；只有在单智能体确实无法胜任时，才演进为多智能体。

## 1. 什么是智能体

传统软件会自动化既定流程；智能体则能以较高自主性代表用户完成流程。文中将智能体定义为能够独立完成任务的系统，并给出两个必要特征：

1. **LLM 控制工作流。** 模型负责决策、判断任务是否结束、根据失败情况调整行动，并在无法继续时把控制权交还给用户。
2. **可动态使用工具。** 智能体可从外部系统获取信息或执行动作，并依据当前工作流状态选择工具；工具使用必须受明确护栏约束。

因此，单轮问答机器人、单次 LLM 调用或情感分类器不属于智能体：它们没有让 LLM 主导工作流的执行。

## 2. 何时值得构建智能体

智能体并非传统确定性自动化的替代品。优先评估那些原有规则系统难以处理的工作流：

| 高适配信号 | 含义 | 示例 |
| --- | --- | --- |
| **复杂决策** | 需要理解上下文、例外与细微判断 | 客服退款审批 |
| **规则难维护** | 分支和例外不断累积，更新成本高、易出错 | 供应商安全审查 |
| **非结构化数据占主导** | 需要读文档、理解自然语言或与人对话 | 房屋保险理赔处理 |

如果这些条件不明显，规则引擎或普通确定性流程往往更简单、便宜且可靠。

## 3. 智能体的三个基础组件

```text
            明确指令（Instructions）
                       │
用户输入 ──►  模型（Model）  ◄──► 工具（Tools）
                       │
                  输出 / 动作
```

| 组件 | 职责 | 关键要求 |
| --- | --- | --- |
| **模型** | 推理、决策、推动工作流 | 在准确率、延迟、成本之间选择 |
| **工具** | 查询上下文或改变外部系统 | 定义标准化、说明完整、可复用、充分测试 |
| **指令** | 规定行为、流程、边界和护栏 | 清楚、可执行、覆盖常见异常情况 |

### 3.1 模型选择：先建立能力基线，再降本

推荐顺序：

1. 建立评测（evals），确定性能基线；
2. 先用能力最强的模型达到目标准确率；
3. 再将其中的简单环节替换为更小、更快、更便宜的模型，并以评测确认质量仍可接受。

这样可以避免一开始就因模型能力不足而误判架构问题。简单的检索或意图分类通常不需要最强模型；复杂判断（如退款审批）则可能需要更强的推理能力。

### 3.2 工具：让智能体能“取数、行动与编排”

工具扩展了模型的外部能力。即便遗留系统没有 API，也可以通过计算机使用能力操作网页或应用界面。

工具可分为三类：

| 工具类别 | 作用 | 示例 |
| --- | --- | --- |
| **数据工具** | 获取执行任务所需的背景和信息 | 查询交易库、CRM、PDF、网页 |
| **动作工具** | 改变外部系统状态 | 发邮件或短信、更新 CRM、转人工工单 |
| **编排工具** | 将一个专业智能体作为另一个智能体的工具 | 退款、研究、写作智能体 |

设计上应保持统一、可发现、可版本管理。工具数量增加时，先检查名称、参数和说明是否清晰；只有这些措施仍无法解决错误选工具或理解困难的问题，再考虑拆分多个智能体。

### 3.3 指令：把业务知识转成可执行例程

高质量指令能减少歧义，改善决策并降低工作流错误。实务建议：

- 用已有 SOP、客服脚本、政策文档和知识库作为指令来源；
- 将密集规则拆成更小、明确的步骤；
- 每一步写清楚对应的动作或输出，例如“索取订单号”“调用账户查询 API”；
- 预设常见边界情况，如信息缺失、问题偏离预期时应如何分支；
- 可先让高能力模型把原始政策文档转换成无歧义的编号指令，再由人工审校。

## 4. 编排：从单智能体开始

文中强调，客户通常通过渐进式架构取得更好效果。无论单智能体还是多智能体，核心都有一个持续运行的循环（run）：模型调用工具、查看结果、继续行动，直到输出结果、发生错误、达到最大轮数，或满足其他退出条件。

### 4.1 单智能体：默认选择

一个拥有合适工具和指令的智能体能够处理许多任务。逐步增加工具能扩展能力，同时保持架构简单，方便评测和维护。

还可用**提示模板**控制复杂度：建立一个携带政策变量的基础提示，而非为每类场景维护一套完整提示。新场景通过替换变量适配，降低维护和评测成本。

### 4.2 何时拆成多智能体

“更多智能体”带来概念分离的好处，也带来协同、上下文与维护成本。出现下列现象时才值得拆分：

- **逻辑过于复杂**：一个提示中塞满多层 if/then/else 分支，提示模板已难扩展；
- **工具负载过高**：模型持续混淆相似或重叠的工具；先完善工具名称、参数和描述，若仍无改善再拆分。

工具的数量并非唯一指标。十几个定义清楚、差异明显的工具可能运行良好；少量彼此重叠的工具也可能造成严重混淆。

### 4.3 两种多智能体模式

| 模式 | 控制方式 | 适合情况 |
| --- | --- | --- |
| **经理模式（Manager / agents as tools）** | 中心“经理”智能体通过工具调用委派专业智能体，并统一整合结果 | 需要一个智能体保持对用户和整体流程的控制 |
| **去中心化交接（Handoff）** | 对等智能体将控制权单向移交给下一个专业智能体，并传递对话状态 | 分诊或明确的专业场景，接手者可直接继续与用户交互 |

前者强调集中协调和统一体验；后者强调由最合适的专业角色直接接手任务。无论哪种模式，组件仍应可组合，并使用结构清晰的提示。

## 5. 护栏：分层安全，而不是单点防护

护栏用来控制隐私、越权、品牌和安全风险，但不能替代认证、授权、访问控制和常规软件安全措施。可靠做法是多层、专用防线叠加：单一护栏通常不足以抵御所有风险。

### 常见护栏类型

| 类型 | 目标 |
| --- | --- |
| **相关性分类器** | 拦截偏离业务范围的问题 |
| **安全分类器** | 识别越狱、提示注入、套取系统提示等攻击 |
| **PII 过滤器** | 防止模型输出不必要的个人身份信息 |
| **内容审核** | 识别仇恨、骚扰、暴力等有害内容 |
| **工具防护** | 按只读/写入、可逆性、权限和财务影响给工具分级；高风险动作需额外检查或转人工 |
| **规则式防护** | 黑名单、输入长度限制、正则表达式、SQL 注入过滤等确定性防线 |
| **输出校验** | 以提示和内容检查确保输出符合品牌和业务要求 |

### 构建策略

1. 首先解决数据隐私和内容安全风险；
2. 根据真实失败案例与边界情况逐步增加护栏；
3. 同时优化安全性和用户体验，随系统演进调整规则。

部分护栏可与主智能体并发运行，在违反约束时中断流程；这种“乐观执行”能降低常态延迟，但高风险动作仍需设置更严格的前置检查。

## 6. 人工介入是系统能力的一部分

人工接管并非失败，而是使早期系统安全上线、发现缺陷并建立评测循环的关键机制。应让智能体在无法完成任务时优雅转交给人工客服或最终用户。

典型触发条件：

- **超过失败阈值**：如多次重试后仍无法理解用户意图；
- **高风险动作**：敏感、不可逆或影响重大的操作，例如取消订单、大额退款、付款授权。

## 7. 推荐的实施路线

```text
1. 选一个确实难以用规则自动化的工作流
              ↓
2. 用强模型、明确指令和少量高质量工具构建单智能体基线
              ↓
3. 用评测和真实用户反馈验证准确率、成本、延迟和安全性
              ↓
4. 用小模型替换可降级的环节，并补齐工具与指令的边界情况
              ↓
5. 按风险分层添加护栏、退出条件与人工接管机制
              ↓
6. 仅当逻辑或工具复杂度已成为瓶颈时，再采用多智能体编排
```

## 8. 最终要点

- 智能体的价值在于对复杂、模糊、依赖非结构化信息的端到端工作流进行决策和行动。
- 可靠性来自基础能力：合适的模型、定义清楚的工具、结构化指令和持续评测。
- 架构应渐进演化：**先单智能体，后多智能体；先验证价值，后扩展自治。**
- 护栏必须贯穿输入、输出、工具调用和人工监督，并与标准安全控制共同工作。
- 最稳妥的部署方式是小范围起步，在真实使用中验证与修正，再逐渐扩大能力。
